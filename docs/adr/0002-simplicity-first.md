# ADR-0002: 简单优先

## 状态

已接受

## 背景

新手常犯的错误：

- 担心"以后可能要用"，提前写很多代码
- 追求"优雅"的抽象，写出复杂的继承层次
- 配置项、参数、选项比实际代码还多

这些代码大部分永远不会用到，却增加了理解和维护的成本。

## 决策

只写解决当前问题需要的最少代码。

**规则：**
1. 不写"以后可能用到"的功能
2. 不为单一用途创建抽象
3. 不添加未请求的配置项
4. 能用3行代码解决的，不写10行

## 原因

1. **YAGNI原则**（You Ain't Gonna Need It）：你预想的需求，80%不会发生
2. **代码是负债**：每一行代码都需要理解、维护、测试
3. **简单容易改**：代码越简单，以后修改越容易

## 怎么做

写代码前问自己：

- 这行代码是为了解决什么问题？
- 如果删掉这行，会怎样？
- 有没有更简单的方式？

**判断标准：** 如果一个资深工程师看到你的代码说"这也太复杂了"，那就是太复杂了。

## 给新手的话

代码不是越多越好。想象你在打包旅行：

- 新手：带5双鞋，万一下雨呢？万一是正式场合呢？
- 老手：带1双最舒服的鞋，其他需要时再买

代码也一样，现在需要什么就写什么，"以后"的事等"以后"再说。
