# ADR-0001: 垂直切片开发

## 状态

已接受

## 背景

传统开发方式是"水平切片"：先写完所有测试，再写完所有实现。这种方式的问题：

- 测试是批量写的，往往测试的是"想象中的行为"，不是真实需求
- 实现也是批量写的，容易偏离测试
- 中间没有反馈，最后才发现问题

## 决策

采用"垂直切片"开发方式：

```
错误（水平）：
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

正确（垂直）：
  RED->GREEN: test1->impl1
  RED->GREEN: test2->impl2
  RED->GREEN: test3->impl3
```

每次只写一个测试，然后只写让这个测试通过的代码。

## 原因

1. **即时反馈**：每个切片都是完整的闭环，立刻知道对不对
2. **减少浪费**：不会写一堆最后用不上的测试
3. **降低风险**：小步走，错了立刻发现

## 怎么做

1. 写一个测试（描述一个具体的需求）
2. 运行测试，确认失败（证明测试有效）
3. 写最少的代码让测试通过
4. 确认测试通过
5. 回到步骤1，写下一个测试

## 给新手的话

想象你在盖房子：

- 水平切片是：先把所有房间的框架搭好，再装所有门窗，再刷所有墙
- 垂直切片是：一个房间一个房间地完成，每个房间都能住人

垂直切片的好处是，你随时可以停下来，已有的部分都是可用的。
