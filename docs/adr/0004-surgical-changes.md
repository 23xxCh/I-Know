# ADR-0004: 精准修改

## 状态

已接受

## 背景

新手常犯的错误：

- 顺手改一下旁边的代码格式
- 看到觉得"不优雅"的代码，顺手重构
- 删掉觉得"没用"的注释

结果：代码审查时改动太多，难以审查；或者改坏了不相关的功能。

## 决策

修改代码时：

1. **只改你该改的**：每一行改动都要能追溯到用户需求
2. **保持现有风格**：即使你不喜欢，也要和周围代码保持一致
3. **不要顺手改进**：改Bug就改Bug，不要顺便重构
4. **清理自己的垃圾**：你改动产生的无用代码要删掉

## 原因

1. **改动越少，出错越少**：每一行改动都有可能引入Bug
2. **代码审查友好**：改动越少，越容易被审查
3. **责任清晰**：出了问题，容易定位是谁改坏的

## 怎么做

**检查清单：**
- [ ] 我改的每一行都是解决用户需求必须的吗？
- [ ] 我有没有顺便改了不相关的东西？
- [ ] 我有没有保持周围的代码风格？

**唯一例外：** 你改的代码产生了无用的导入、变量、函数——这些要删掉，因为是你制造的垃圾。

## 给新手的话

想象你在修复一幅画：

- 错误做法：修了一个角，顺手把旁边的颜色也改了
- 正确做法：只修需要修的地方，其他地方不要动

代码也是一样，不要"顺手"改东西。如果真的需要改别的地方，那是另一个任务，另外做。
