# ADR-0003: 先思考后编码

## 状态

已接受

## 背景

新手常犯的错误：

- 需求还没说清楚，就开始写代码
- 有多种可能的理解，自己选一个就开始做
- 遇到不确定的地方，猜一个答案继续

结果：做出来的东西不是用户要的，浪费时间重写。

## 决策

在开始编码之前，必须确保需求清晰：

1. **明确假设**：如果你有任何"应该是..."的想法，先确认
2. **识别歧义**：如果有多种理解，列出来让用户选
3. **暴露权衡**：如果A和B不能兼得，告诉用户

**Grill模式**：需求不清晰时，问2-4个问题，而不是开始编码。

## 原因

1. **改代码比写代码贵**：写一行代码1分钟，改一行代码可能要1小时
2. **错误会放大**：早期理解错误，后面所有代码都是错的
3. **用户也不清楚**：用户说需求时也没想清楚，需要你帮他理清

## 怎么做

**错误示范：**
> 用户：帮我加个登录功能
> AI：好的，开始写代码...

**正确示范：**
> 用户：帮我加个登录功能
> AI：等等。先回答几个问题：
>   1. 这是什么类型的应用？Web？App？
>   2. 需要第三方登录吗？还是只要用户名密码？
>   3. 安全要求高吗？需要二次验证吗？

## 给新手的话

想象你在做菜：

- 不问清楚就做：客人想要清淡，你做了麻辣，白做了
- 先问清楚再做：确认口味再下锅，一次成功

写代码比做菜贵多了，不要怕问问题。
