# 拆分 Issues

## 核心理念

**大任务拆成小任务。小任务才能执行。**

一个"实现用户系统"的需求太大，没法直接动手。
拆成"注册、登录、找回密码、个人信息"这些小任务，每个都能独立完成。

---

## 拆分原则

### 原则1：垂直切片

**不要水平切片！**

```
错误（水平）：
  Issue 1：写所有测试
  Issue 2：写所有后端
  Issue 3：写所有前端

正确（垂直）：
  Issue 1：注册功能（测试+后端+前端）
  Issue 2：登录功能（测试+后端+前端）
  Issue 3：找回密码功能（测试+后端+前端）
```

**为什么垂直切片：**
- 每个 Issue 是完整的功能
- 可以独立开发、测试、部署
- 不用等所有东西都做完才能用

### 原则2：独立可完成

每个 Issue 应该：
- 能独立开发
- 能独立测试
- 能独立部署（可选）
- 不依赖其他未完成的 Issue

**检查方式：**
```
问自己：如果只有这个 Issue，能完成吗？

能 → 独立性好
不能 → 拆得不对，或者依赖太多
```

### 原则3：粒度适中

**太大：**
- "实现用户系统"（太大，无法下手）
- "重构代码库"（太大，无法验证）

**太小：**
- "添加一行代码"（太小，没有意义）
- "改一个变量名"（太小，应该包含在其他 Issue 中）

**正好：**
- "实现用户注册功能"（一个完整功能）
- "重构登录模块"（一个明确范围）

**判断标准：**
- 一个开发者能在1-3天内完成
- 有明确的验收标准
- 能用一个标题说清楚

## 拆分流程

### 第一步：理解整体需求

先搞清楚要做什么：

```
需求：实现用户系统

包含什么？
- 用户注册
- 用户登录
- 找回密码
- 个人信息管理
- 账号注销
```

### 第二步：识别核心路径

找出"最小可用"的部分：

```
用户系统的核心路径：
注册 → 登录 → 使用功能

没有注册和登录，其他都没意义。

所以先拆出：
1. 用户注册
2. 用户登录
```

### 第三步：逐层拆分

对每个部分继续拆分：

```
用户注册：
├── 基础注册（邮箱+密码）
├── 邮箱验证
├── 第三方注册（Google、微信）
└── 注册引导

用户登录：
├── 基础登录（邮箱+密码）
├── 记住登录
├── 第三方登录
└── 登录失败处理
```

### 第四步：写出 Issue

**Issue 模板：**

```markdown
## 标题
[简短描述，动词开头]

## 背景
[为什么做这个？解决什么问题？]

## 需求
- [具体要做什么]
- [功能点1]
- [功能点2]

## 验收标准
- [ ] [怎么算完成]
- [ ] [测试通过]
- [ ] [可以演示]

## 技术方案（可选）
[如果有明确的技术方案，写在这里]

## 依赖
- [依赖的其他 Issue，如果没有就写"无"]
```

## 示例

### 需求

```
实现一个用户注册功能
```

### 拆分后的 Issues

**Issue 1：基础注册功能**

```markdown
## 标题
实现基础用户注册（邮箱+密码）

## 背景
用户需要注册账号才能使用系统的个性化功能。

## 需求
- 用户可以用邮箱和密码注册
- 密码至少8位，包含字母和数字
- 邮箱格式验证
- 邮箱不能重复

## 验收标准
- [ ] 用户可以用邮箱+密码注册成功
- [ ] 重复邮箱提示"邮箱已被注册"
- [ ] 密码强度不够时提示要求
- [ ] 邮箱格式错误时提示

## 依赖
无
```

**Issue 2：邮箱验证**

```markdown
## 标题
实现注册邮箱验证

## 背景
确保用户填写的邮箱是真实可用的。

## 需求
- 注册后发送验证邮件
- 邮件包含验证链接
- 点击链接完成验证
- 验证链接24小时有效

## 验收标准
- [ ] 注册后自动发送验证邮件
- [ ] 点击链接后账号状态变为"已验证"
- [ ] 过期链接提示"链接已过期"

## 依赖
- Issue 1：基础注册功能
```

## 常见错误

### 错误1：拆分太细

```
错误：
Issue 1：添加注册表单 HTML
Issue 2：添加注册表单 CSS
Issue 3：添加注册表单 JS
Issue 4：写注册 API
Issue 5：写数据库存储

正确：
Issue 1：基础注册功能（包含以上所有）
```

### 错误2：拆分太粗

```
错误：
Issue 1：用户系统

正确：
Issue 1：用户注册
Issue 2：用户登录
Issue 3：找回密码
Issue 4：个人信息管理
```

### 错误3：依赖循环

```
错误：
Issue A 依赖 Issue B
Issue B 依赖 Issue A

正确：
重新设计，打破循环
或者找出真正的优先级
```

## 给新手的话

拆分 Issue 就像切蛋糕：

- 不要把整个蛋糕一次吃完（太大）
- 不要切成碎屑（太小）
- 切成一口大小的块（刚好）

一个好的 Issue：
- 一个人能在1-3天完成
- 有明确的开始和结束
- 能独立验证
