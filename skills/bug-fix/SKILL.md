# Bug 修复

## 核心理念

**没有复现步骤，不修Bug。**

猜测是时间的杀手。如果你不能稳定复现一个Bug，你就无法验证你修好了它。

---

## 工作流程

### 第一步：收集信息

**问用户这些问题：**

| 问题 | 为什么重要 |
|------|-----------|
| 错误信息是什么？ | 直接指向问题 |
| 你做了什么操作？ | 理解触发条件 |
| 预期结果是什么？ | 理解正确行为 |
| 实际结果是什么？ | 理解错误行为 |
| 每次都发生吗？ | 理解是否稳定 |

**如果用户说"不知道怎么复现"：**
```
理解。让我们一起来找：

1. 你最后一次看到这个Bug是什么时候？
2. 当时你在做什么？
3. 最近有什么改动？

如果你能给我一个大概的场景，我可以帮你定位。
```

### 第二步：复现Bug

**找到稳定复现的步骤：**

```
复现步骤：
1. 打开页面 X
2. 点击按钮 Y
3. 输入 Z
4. 结果：[错误]

预期结果：[正确行为]
实际结果：[错误行为]
```

**为什么必须复现：**
- 复现 = 理解问题
- 不能复现 = 瞎猜
- 能复现才能验证修复

### 第三步：最小化

把复现步骤简化到最简：

- 删掉无关代码
- 删掉无关输入
- 找到触发条件

**最小化的好处：**
- 问题范围缩小，容易定位
- 干扰因素减少，逻辑清晰
- 测试更简单

### 第四步：写失败的测试

```python
def test_bug_xxx():
    """
    Bug: [描述]
    复现步骤：[步骤]
    """
    # 这个测试目前会失败
    result = do_something()
    assert result == expected  # 失败，因为Bug存在
```

**为什么要写测试：**
- 测试是证据，证明Bug存在
- 测试是安全网，保证修复有效
- 测试是文档，记录这个Bug和修复

### 第五步：假设与验证

**提出假设，设计实验验证：**

```
假设：问题出在 [某处]
验证：[如何检查]
结果：[确认/否定]
```

**验证方法：**

1. **加日志**：在关键位置打印信息
2. **断点调试**：一步一步执行
3. **二分法**：注释掉一半代码，看问题在哪一半
4. **隔离测试**：单独运行有问题的部分

**假设失败的应对：**
```
假设失败，排除 [某处]。

新假设：问题可能在 [另一处]。
验证方法：[新的检查方式]
```

### 第六步：修复

**找到根因后修复。**

**修复原则：**
- 只修这个Bug
- 不要顺手"改进"其他地方
- 改动最小化
- 保持代码风格一致

### 第七步：回归测试

**确保修复没有引入新问题：**

- [ ] Bug测试通过（证明修复有效）
- [ ] 所有原有测试通过（证明没有破坏其他功能）
- [ ] 手动验证（让用户确认问题解决）

## 常见问题

### Q: 用户说"很急，直接改吧"

**回应：**
```
理解，很急。

但让我先花1分钟确认一下问题，避免改错地方更浪费时间。

[问1-2个关键问题]
```

### Q: 改了一个地方，另一个地方坏了

**回应：**
```
这说明两件事：
1. 原来的修复可能不对，影响范围比预期大
2. 或者是另一个问题被触发了

让我检查一下：
- 这个改动影响了哪些代码？
- 有没有测试覆盖这些地方？

我们可能需要先写更多测试，再继续修复。
```

### Q: 找不到问题原因

**回应：**
```
确实有些问题很难定位。让我试试其他方法：

1. [方法1] - 比如：检查最近的改动
2. [方法2] - 比如：对比正常和异常的数据
3. [方法3] - 比如：简化问题场景

如果这些都不行，可能需要更多上下文信息：
- 这个问题是什么时候开始的？
- 最近有什么大的改动？
```

## 调试技巧

### 技巧1：二分法

```
问题：代码有Bug，不知道在哪

方法：
1. 注释掉一半代码
2. 问题还在？Bug在剩下的一半
3. 问题没了？Bug在注释掉的一半
4. 重复，直到找到具体位置
```

### 技巧2：日志调试

```
在关键位置加日志：

print("1. 进入函数")
print("2. 参数是：", params)
print("3. 计算结果是：", result)
print("4. 函数结束")

看日志，问题出在哪两步之间。
```

### 技巧3：隔离测试

```
把有问题的代码单独拿出来测试：

def test_isolated():
    # 只测试有问题的部分
    result = problematic_function(simple_input)
    assert result == expected
```

## 给新手的话

修Bug就像医生看病：

1. **收集信息** = 问诊（哪里不舒服？什么时候开始的？）
2. **复现** = 检查（让我看看怎么不舒服）
3. **假设验证** = 化验（可能是X，做个检查确认）
4. **修复** = 治疗（对症下药）
5. **回归测试** = 复查（确认治好了，没有副作用）

不要还没诊断就开刀。确认问题再动手。
