# 代码概览（Zoom Out）

## 核心理念

**看代码要看森林，不是只看树木。**

当你只盯着一个函数看，容易迷失在细节里。
退后一步，看看：
- 这个模块在整个系统里的位置
- 谁在调用它
- 它依赖谁

---

## 概览输出

当你请求概览时，我会给你：

```
## 模块位置
[这个模块在系统里的角色]

## 上游调用者
[谁调用了这个模块]

## 下游依赖
[这个模块依赖谁]

## 关键数据流
[数据怎么流过这个模块]

## 核心文件
[最重要的几个文件]
```

## 使用场景

### 场景1：接手别人的代码

```
用户：我刚接手这个项目，这是什么东西？

我：让我给你一个概览：

## 项目定位
这是一个电商平台的后端服务，负责订单处理。

## 系统架构
用户请求 → API层 → 服务层 → 数据层 → 数据库

## 核心模块
- orders/ - 订单核心逻辑
- payments/ - 支付处理
- inventory/ - 库存管理
- notifications/ - 通知服务

## 关键数据流
用户下单 → 检查库存 → 创建订单 → 处理支付 → 发送通知

## 建议阅读顺序
1. README.md
2. docs/architecture.md
3. orders/service.py（入口文件）
```

### 场景2：定位 Bug

```
用户：这个函数出Bug了，但我不知道谁在调用它

我：让我查一下调用关系：

## 函数位置
processPayment() 在 payments/service.py

## 调用者（上游）
- orders/checkout.py:127 - 结账时调用
- subscriptions/renewal.py:45 - 续费时调用
- refunds/process.py:23 - 退款时调用

## 被调用者（下游）
- stripe_api.charge() - 调用支付网关
- transactions/log() - 记录交易日志
- inventory/decrease() - 减库存

## 你的Bug可能涉及
- checkout.py 是最常见的调用路径，从这里开始查
```

### 场景3：重构前调研

```
用户：我想重构这个模块，会影响什么？

我：让我分析一下影响范围：

## 模块：user_auth

## 直接影响
- 15个文件调用了这个模块
- 3个API端点依赖它

## 调用者列表
1. api/login.py - 登录接口
2. api/register.py - 注册接口
3. api/profile.py - 个人信息接口
4. middleware/auth.py - 认证中间件
... 还有11个

## 建议
- 这是一个核心模块，改动要小心
- 先写好测试覆盖
- 分阶段重构，每阶段验证
```

## 概览类型

### 类型1：模块概览

了解一个模块在系统中的位置。

```
/zoom-out auth/
```

输出：
- 模块职责
- 对外接口
- 内部结构

### 类型2：函数概览

了解一个函数的调用关系。

```
/zoom-out processPayment
```

输出：
- 调用者
- 被调用者
- 数据流

### 类型3：文件概览

了解一个文件的结构。

```
/zoom-out orders/service.py
```

输出：
- 文件职责
- 主要函数/类
- 依赖关系

---

## 给新手的话

看代码就像看地图：

- 不要只盯着一条街看（一个函数）
- 退后一步，看整个城市（系统架构）
- 看这条路连接了哪些地方（调用关系）

当你理解了"这个模块在做什么"，再看细节就容易多了。
