# 系统化调试

## 核心理念

**没有复现，不修Bug。没有假设，只有证据。**

瞎猜是时间的杀手。看到问题不要马上改代码，先建立反馈循环。

---

## 调试流程

### 第一步：建立反馈循环

**这是最关键的一步。没有反馈循环，一切都是瞎猜。**

什么是反馈循环？
- 一个能告诉你"问题还在不在"的方法
- 越快越好（几秒钟，不是几分钟）
- 越确定越好（100%复现，不是"有时候"）

**反馈循环的类型：**

| 类型 | 例子 | 速度 |
|------|------|------|
| 失败的测试 | `npm test -- --grep "login"` | 秒级 |
| 命令行脚本 | `node repro.js` | 秒级 |
| UI 操作 | 点击按钮 → 看到404 | 分钟级 |
| 日志监控 | `tail -f error.log` | 实时 |

**如果你没有反馈循环：**
```
等等。你现在不能修这个Bug。

为什么？因为你不知道：
1. 问题到底是什么
2. 改完后有没有修好

先花5分钟建立一个能复现问题的方法。
这5分钟会省你几小时。
```

### 第二步：最小化

把问题缩小到最小范围：

1. **删掉无关代码**：注释掉一半，问题还在吗？
2. **删掉无关输入**：哪个参数触发了问题？
3. **找到触发条件**：什么情况下必然发生？

**二分法：**
```
问题：1000行代码有Bug

步骤：
1. 注释掉后500行
2. 问题还在？Bug在前500行
3. 问题没了？Bug在后500行
4. 重复，直到找到具体位置
```

### 第三步：提出假设

**假设 = 你认为问题在哪**

```
假设：问题出在 [具体位置]
因为：[观察到的证据]
验证方式：[怎么确认]
```

**好的假设：**
- 具体（不是"代码有问题"，而是"第47行返回了undefined"）
- 可验证（有明确的验证方式）
- 有证据支持（基于观察，不是猜测）

### 第四步：验证假设

**验证方法：**

1. **加日志**
   ```
   在关键位置加：
   console.log("1. 进入函数", params)
   console.log("2. 查询结果", result)
   console.log("3. 返回值", returnValue)
   ```
   看日志，问题出在哪两步之间。

2. **断点调试**
   - 在可疑位置打断点
   - 一步一步执行
   - 观察变量值的变化

3. **隔离测试**
   ```python
   def test_isolated():
       # 只测试有问题的部分
       result = problematic_function(simple_input)
       assert result == expected
   ```

### 第五步：修复

找到根因后，**只修这个问题**。

- 不要顺手重构
- 不要改其他地方
- 改动最小化

### 第六步：验证修复

- [ ] Bug 不再复现
- [ ] 没有引入新问题
- [ ] 所有测试通过

**写回归测试：**
```python
def test_bug_xxx_fixed():
    """确保这个Bug永远不会再出现"""
    # 之前会失败的场景
    result = do_something(bad_input)
    assert result.success  # 现在通过了
```

## 常见错误

### 错误1：没有复现就动手

```
用户：有个Bug，有时候会崩溃

我：等等。"有时候"是多少？
   - 10次里1次？
   - 特定条件下？
   - 完全随机？

不能稳定复现的Bug是修不好的。
先找到必现的步骤。
```

### 错误2：改多处地方

```
错误：我觉得可能是这里的问题，改一下试试...
     好像没好，再改那里试试...
     改了5个地方，终于好了...但不知道是哪个修好的

正确：找到根因 → 只改这一处 → 验证修复
```

### 错误3：假设太多

```
错误：可能是A的问题，也可能是B，也可能是C...
     我先把A、B、C都改了

正确：一个假设，一次验证。不行就下一个假设。
```

## 调试工具箱

### 1. 日志调试

最简单、最通用：

```javascript
// 在关键位置加日志
console.log("=== DEBUG START ===")
console.log("input:", input)
console.log("result:", result)
console.log("=== DEBUG END ===")
```

### 2. 二分法

快速定位问题位置：

```
1. 注释掉一半代码
2. 问题还在？在剩下的一半
3. 问题没了？在注释掉的一半
4. 重复直到找到
```

### 3. 橡皮鸭调试

对着"橡皮鸭"（或者空气）解释你的代码：

```
"这里我做了X，期望结果是Y，但实际是Z..."
说着说着，你可能就发现问题了。
```

### 4. 简化输入

找到触发问题的最小输入：

```
错误：输入一个复杂的JSON，出问题了

正确：
- 删掉一半字段，还有问题吗？
- 再删一半...
- 找到最小触发条件
```

## 给新手的话

调试就像破案：

1. **收集证据**：错误信息、日志、复现步骤
2. **列出嫌疑人**：可能出问题的地方
3. **逐一排查**：验证每个假设
4. **找到真凶**：确定根本原因
5. **结案**：修复并验证

不要看到问题就改代码。先调查，后行动。
