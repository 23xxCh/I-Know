# 需求访谈（Grill Me）

## 核心理念

**需求没说清楚，不动手。**

用户说"帮我做一个登录功能"，这根本不是需求，只是一个想法。

动手之前，必须搞清楚：
- 到底要做什么？
- 为什么做这个？
- 怎么算完成？

---

## 访谈流程

### 第一轮：搞清楚"是什么"

**必须回答的问题：**

| 问题 | 为什么重要 |
|------|-----------|
| 这个功能解决什么问题？ | 理解目标 |
| 用户是谁？ | 理解场景 |
| 用户怎么使用它？ | 理解交互 |
| 输入是什么？输出是什么？ | 理解边界 |

**话术：**
```
等等，在你开始之前，我需要确认几个问题：

1. 这个功能是给谁用的？
2. 他们会在什么场景下使用？
3. 输入是什么？输出是什么？
4. 有什么边界情况需要处理？

这些不回答清楚，做出来的东西可能不是你想要的。
```

### 第二轮：搞清楚"为什么"

**追问：**
- 为什么需要这个功能？
- 有没有现成的解决方案？
- 不做这个会怎样？

**目的：** 确认这个需求真的值得做。

### 第三轮：搞清楚"怎么做"

**技术细节：**
- 有什么技术限制？
- 需要兼容什么？
- 性能要求是什么？
- 安全要求是什么？

### 第四轮：确认理解

**复述需求：**
```
让我确认一下我理解的：

你想要 [功能描述]，给 [用户类型] 使用。
输入是 [X]，输出是 [Y]。
边界情况包括 [Z]。
成功标准是 [W]。

对吗？
```

## 问问题的技巧

### 开放式问题 vs 封闭式问题

**封闭式（选择题）：**
- "是用邮箱登录还是手机号登录？"
- "需要第三方登录吗？"

**开放式（问答题）：**
- "这个功能是解决什么问题的？"
- "用户怎么使用它？"

**技巧：** 先用开放式问题了解全貌，再用封闭式问题确认细节。

### 一次问多少问题？

**错误：**
```
我需要确认几个问题：
1. ...
2. ...
3. ...
4. ...
5. ...
6. ...
7. ...
（用户看到就累了）
```

**正确：**
```
等等，先回答几个问题：
1. 这个功能是给谁用的？
2. 他们怎么使用它？

（用户回答后）
好，还有几个问题：
3. 输入是什么？输出是什么？
4. 有什么特殊情况？
```

**原则：** 每次2-4个问题，不要一次丢太多。

## 常见场景

### 场景1：需求模糊

```
用户：帮我做一个登录功能

我：等等。先回答几个问题：

    1. 这是什么类型的应用？Web？App？
    2. 用户用什么登录？邮箱？手机号？用户名？
    3. 需要第三方登录吗？微信？Google？
    4. 安全要求高吗？需要二次验证吗？

    这些问题不回答，做出来的东西一定是错的。
```

### 场景2：需求不合理

```
用户：我要做一个万能导入功能，能导入Excel、CSV、JSON、XML...

我：等等。让我确认一下：

    你真的需要同时支持这么多格式吗？
    - 用户会传Excel吗？多久一次？
    - CSV和JSON是不是就够了？

    如果用户一年才传一次Excel，值得为此开发吗？

    我建议先支持CSV和JSON，看看是否够用。
    不够再加Excel。你觉得呢？
```

### 场景3：需求冲突

```
用户：登录要简单，但也要安全，最好刷脸登录

我：这里有冲突：

    简单 = 用户少操作
    安全 = 用户多验证
    刷脸 = 需要摄像头，不是所有设备都有

    你更看重哪个？
    A) 简单优先：邮箱+密码，可选记住登录
    B) 安全优先：邮箱+密码+验证码
    C) 平衡：邮箱+密码，敏感操作需要二次验证
```

---

## 完成标准

访谈完成的标志：

- [ ] 知道做什么（功能描述）
- [ ] 知道为什么做（目标）
- [ ] 知道怎么做（技术方案）
- [ ] 知道怎么算完成（成功标准）
- [ ] 用户确认理解正确

---

## 给新手的话

Grill Me 就像医生问诊：

- 医生不会你说"头疼"就开药
- 医生会问：哪里疼？多久了？有没有其他症状？
- 问清楚了，才能对症下药

需求也一样。说"做一个登录功能"就像说"头疼"。
问清楚细节，才能做对东西。

**记住：问问题不是拖延，是避免做错重来。**
